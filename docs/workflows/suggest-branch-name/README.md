# Branch Name Suggestion

## 概要

現在の git の変更内容を分析し、ブランチ命名規則に従って最も適切なブランチ名を決定する。main ブランチで作業している場合は、その名前で新しいブランチを作成して自動的に切り替える。

## 呼び出し方法

| 環境 | コマンド |
|------|----------|
| Claude Code (CLI) | `/suggest-branch-name` |
| Codex | `$suggest-branch-name` |

## 手順

1. 現在のブランチを確認する
2. git status と git diff を確認して、変更内容を分析する
3. 変更の意図（新機能、バグ修正、リファクタなど）を判断する
4. ブランチ命名規則に従って、最も適切なブランチ名を 1 つ決定する
5. 補足として代替候補を 2 つまで挙げる
6. main ブランチにいる場合は、決定したブランチ名で新しいブランチを作成する

## リポジトリの確認

まず現在の状態を把握する。

```bash
# 現在のブランチを確認
git branch --show-current

# 変更内容のサマリーを確認
git status --short

# 変更内容の詳細を確認
git diff --stat

# 実際の差分を確認（必要に応じて）
git diff
```

変更がない場合は、その旨を伝えて終了する。

## 変更内容の分析

以下の観点で変更を分析する:

- **追加されたファイル**: 新機能の追加か、既存機能の拡張か
- **削除されたファイル**: 不要な機能の削除か、リファクタの一部か
- **変更されたファイル**: バグ修正か、機能改善か、リファクタか
- **影響範囲**: 特定の機能領域か、横断的な変更か
- **ファイルの種類**: ソースコード、テスト、ドキュメント、設定ファイルなど

## ブランチ名の形式

`<type>/<description>` の形式を使う。`description` は kebab-case の英語とする。

### type の種類

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `refactor` | リファクタ（振る舞いを変えない改善） |
| `docs` | ドキュメントのみの変更 |
| `chore` | ビルド設定・依存関係など雑務 |
| `test` | テストのみの変更 |
| `remove` | 機能の削除 |

### description の決め方

- **具体的で分かりやすい**: 何を変更したかが一目で分かる
- **簡潔**: 3〜5 単語程度
- **kebab-case**: 単語はハイフンで区切る
- **英語**: 日本語は使わない

### 良い例

- `feat/user-authentication` - ユーザー認証機能の追加
- `fix/api-base-url-xcconfig` - API base URL の設定バグ修正
- `refactor/settings-view-structure` - 設定画面の構造リファクタ
- `remove/appearance-settings` - 外観設定機能の削除
- `docs/update-review-guidelines` - レビューガイドラインの更新
- `feat/bluetooth-device-scanning` - Bluetooth デバイススキャン機能
- `fix/notification-permission-flow` - 通知権限フローの修正

### 避けるべき例

- `update-stuff` - 曖昧で何をしたか分からない
- `fix-bug` - どのバグか分からない
- `my-branch` - 変更内容が分からない
- `feat/add-new-feature` - 冗長で具体性がない

## 提案の方法

変更内容を分析したら、最有力のブランチ名を 1 つ決定する。必要なら補足として 2 つまで代替案を挙げる。

1. **採用するもの**: 変更の本質を最もよく表すブランチ名
2. **代替案 1**: 別の観点から見た適切なブランチ名
3. **代替案 2**: さらに別の観点から見たブランチ名（必要に応じて）

各提案には、以下を含める:

- ブランチ名
- そのブランチ名を選んだ理由
- どの変更内容を重視したか

## 自動採用ルール

ユーザー確認はデフォルトでは行わない。分析結果から最も適切と判断したブランチ名をその場で採用する。

- main ブランチにいる場合: 採用したブランチ名で `git checkout -b <branch>` を実行する
- main 以外のブランチにいる場合: 現在のブランチを維持し、必要なら採用候補を報告する
- ユーザーが明示的に候補比較や確認を求めた場合のみ、複数案を提示して選んでもらう

## ブランチの作成

現在 main ブランチにいる場合のみ、決定したブランチ名で新しいブランチを作成する。

```bash
git checkout -b <選択されたブランチ名>
```

main 以外のブランチにいる場合は、現在のブランチのままにし、採用候補のみ報告する。ブランチ名変更が必要ならユーザーの明示的な依頼がある場合だけ対応する。

## 結果の報告

完了後は以下を報告する:

- 分析した変更内容のサマリー
- 採用したブランチ名とその理由
- 代替案（ある場合のみ）
- 作成したブランチ名（作成した場合）
- 次に推奨するアクション（コミット作成、PR 作成など）

## 注意事項

- **推測しない**: git status と git diff の結果に基づいて判断する
- **最有力を決める**: 複数候補を並べるだけで終わらず、必ず採用案を 1 つ決める
- **理由を説明**: なぜそのブランチ名を採用したかを説明する
- **main ブランチへの直接コミットを防ぐ**: main にいる場合は必ず採用案でブランチを作成する
